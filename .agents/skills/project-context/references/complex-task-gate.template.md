# 复杂任务复述门控

> **完整方法论** → [AI-COLLABORATION-CHARTER.md](AI-COLLABORATION-CHARTER.md) §一  
> 本文件 = **可检查的硬触发细则**

## 1. 为什么要「硬」一点

仅靠 AGENTS 里的「自动识别」**不能保证**每轮都复述。  
本门控用 **关键词 + 固定首段标题**，维护者一眼能验：**没出现 `## 复述需求` = 流程未执行**。

**诚实边界**：Cursor **无法** 100% 阻止 AI 跳过复述；硬触发 = **规则 + gotchas 索引 + 你可验收的格式**。

## 2. 触发条件（命中任一 → 启用门控）

### 2.1 任务类型

- 架构 / 数据流 / 多系统
- **新增路由或改信息架构**
- 不可逆或高返工
- 需求跨多条线
- 维护者说：**探讨、评估、规划、矩阵、盘点、确认你真的知道、方案、先写计划**

### 2.2 用户话术关键词（粗匹配）

```
探讨|评估|规划|矩阵|盘点|复述|确认.*知道|方案|计划|信息架构|IA|页面.*功能|该不该|什么时候做
```

## 3. 命中后：首条回复**必须**以此开头

```markdown
## 复述需求

（用你自己的话：目标 / 约束 / 非目标；3～8 行或表格）

**非目标（本条不做）**：（若有）

---
```

之后才允许：写文件、改代码、给长方案。

**例外**：维护者明确说「不用复述，直接做」。

## 4. 与四步的衔接

| 步 | 门控要求 |
|----|----------|
| 1 复述 | `## 复述需求` **必填** |
| 2 信息清单 | 复述后同一回复或紧接第二条 |
| 3 完备度声明 | 缺关键项 → **只规划不写码** |
| 4 执行 | 小步试点 |

## 5. 分层实现（推荐）

| 层 | 手段 | 状态 |
|----|------|------|
| **L1** | `project-gotchas.mdc` alwaysApply + 本文件 | 每会话注入 |
| **L2** | `.cursor/hooks` sessionStart 提醒 | 可选 |
| **L3** | stop Hook 检测无复述 | 易扰人，慎用 |

**验收**：首条有没有 `## 复述需求`；没有就回一句「走 COMPLEX-TASK-GATE」。

## 6. 验收清单

- [ ] 首条回复含 `## 复述需求`
- [ ] 有 **非目标**（复杂任务时）
- [ ] 改 IA/路由前先有 **计划或盘点表**，不直接改 `{APP_DIR}/`
