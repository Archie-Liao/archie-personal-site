## 零、背景：为什么要写进一个文件



### 0.1 发生过什么问题

1. **信息散落** — STATUS、AGENTS、guidelines、sessions、代码各说各话；你以为 AI 有一份「始终在更新的网站说明书」，实际没有。
2. **旧需求覆盖新需求时无规程** — 旧 md 像仍有效，新决策只写在对话里，下一会话 AI 又按旧 doc 干活。
3. **复杂任务与判断几乎被删** — 2026-06-30 AI 为「精简 AGENTS」把完整四步与底层方法论压成几行，**严重违背**你的意图。
4. **对话归档不完整** — 用摘要冒充 Archie 原文，轮次顺序反了，还用 Cursor 内部行号（270、310）当序号。



### 0.2 本宪章与项目其他 doc 的分工


| 文档                         | 管什么                          | 不管什么           |
| -------------------------- | ---------------------------- | -------------- |
| **本宪章**                    | **全局可复用的协作规则**（尤其 Archie 主动要求添加的）；写法见 §0.4 | 本项目背景、案例、路径级优先级表 |
| `docs/MULTI-SOURCE-EXEC-PLAN.md` 等 | 本仓库对宪章规则的**项目细则**（路径、案例、任务起点表） | 替代宪章写通用方法论 |
| `docs/SITE-MANUAL.md`      | 网站**现在**几页、什么布局、已实现/规划       | 协作方法论          |
| `docs/STATUS.md`           | 进度、下一步、最近决策                  | 协作四步细则         |
| `AGENTS.md`                | AI 入口 + **索引**（每会话注入）；须链到本宪章 | 不应替代本宪章做唯一细则源  |
| `knowledge/raw/dialogues/` | Archie **原文**语料（女娲蒸馏）        | 规则本身           |




### 0.3 方法论来源

- 在本仓库**实践中自洽生长**；不依赖外部 GitHub skill 抄模板。  
- Gate 0–3 仅 **「知识库 × 网站」** 分支专用，不是全局万能清单。

### 0.4 宪章书写要求

见文首 **「本文件定位」**；改宪章前自检：去掉本仓库路径后，本章是否仍成立？否 → 内容挪专题 doc。

## 一、复杂任务与判断（上下文 · 自检 · 复核）



### 1.1 为什么要这套流程

- 你是建站小白，很多分支（视觉 / 内容 / Obsidian / OKF）**不能拍脑袋同时动**。  
- AI 默认倾向「信息好像够了就开始写代码」——最大风险是**脑内模型不完整仍动手**。  
- 你需要可检查的：**复述、清单、完备度声明、可复核的判断**，而不是「信我」。  
- 这套流程**不是** Cursor 内置能力；是你明确要求写入项目、且**每会话必守**的。



### 1.2 流程 1 · 底层方法论（复杂任务 / 信息是否充足）


| #   | 结论           | 含义                                         |
| --- | ------------ | ------------------------------------------ |
| 1   | **先外化，再行动**  | 复述需求 + 信息清单 = 把隐性假设变成 Archie 可检查的文字。       |
| 2   | **清单随任务变**   | 「缺什么会做错」取决于具体任务；Gate 0–3 只是知识库分支的专用清单。     |
| 3   | **完备度是责任声明** | 必须写清：已满足 / 仍缺 / 缺了会导致什么；**缺关键项 → 只规划不写码**。 |
| 4   | **默认小步试点**   | 即使信息够，也先小范围验证再铺开，便于复核与回滚。                  |




### 1.3 流程 2 · 底层方法论（复杂判断 / 决策可复核）


| #   | 结论                 | 含义                         |
| --- | ------------------ | -------------------------- |
| 1   | **判断要留证据链**        | 日后能回答「为何选 A 不选 B」，可对照事实检验。 |
| 2   | **「不做 / 延后」是合法选项** | 与「现在做」同等认真比较；避免为动而动。       |
| 3   | **量化是对抗直觉偏误**      | 自设维度与权重不是官僚，是让双方对同一标准说话。   |
| 4   | **推荐与假设分离**        | 写清依赖哪些未验证前提；前提不成立时推荐应如何变。  |
| 5   | **时机是决策的一部分**      | 「对不对」与「现在做对不对」是两道题。        |




### 1.4 何时自动视为「复杂任务」（命中任一即启用）

- 架构 / 数据流 / 多系统整合（如网站 + Obsidian + OKF）  
- 新增路由或改变信息架构（IA）  
- 不可逆或高返工成本（内容格式、目录大规模迁移）  
- Archie 说「探讨」「不确定什么时候做」「你定标准」  
- 需求跨多条线（视觉 + 内容 + 知识库同时动）  
- 任务要求来自 **≥2 个权威文件**（Skill、docs、`_spec`、`guidelines` 等）→ 启用 **§1.8 多源执行计划**



### 1.5 复杂任务四步（信息是否够用）— 操作细则


| 步            | 做什么                   | 产出写哪                       | 自检                               |
| ------------ | --------------------- | -------------------------- | -------------------------------- |
| **1 复述需求**   | 用自己的话写清：目标、约束、**非目标** | 对话回复 / 对话归档 / STATUS「最近决策」 | Archie 未反对才可继续；复述也要帮 Archie 理清思路 |
| **2 自定信息清单** | 列 5～10 条「没有则一定做错」的信息  | 同上或 sessions               | 是否按**本任务类型**列的？是否误用 Gate 当全局表？   |
| **3 完备度声明**  | 已满足 / 仍缺 / 缺了会怎样      | 必须明示                       | 若缺**关键项** → **停止写码**             |
| **4 执行或归档**  | 够 → 小步试点；仅探讨 → 只规划    | sessions + STATUS          | 是否偷偷扩大范围？                        |


**知识库 × 网站分支专用 Gate（非全局）**


| Gate | 内容                               |
| ---- | -------------------------------- |
| 0    | 痛点、库存规模、工作流、数据源份数、图谱需求、deadline  |
| 1    | skill 路径、Karpathy/OKF 参考、vault 树 |
| 2    | 样例笔记、字幕、外链、编辑流、build vs 运行时      |
| 3    | 规模、同步方案、写回 vault 含义              |




### 1.6 复杂判断四步（结论要可复核）— 操作细则


| 步             | 做什么                        | 自检                   |
| ------------- | -------------------------- | -------------------- |
| **1 选项 ≥2**   | 必须含「不做 / 延后」               | 是否只给了你想听的那一个？        |
| **2 可量化标准**   | 自列 3～6 维度 + 权重或 1–5 分      | 维度是否与**本判断**相关？      |
| **3 推荐 + 假设** | 写明依赖的未验证前提                 | 推荐与假设是否混在一起？         |
| **4 时机判断**    | 现在做 / 视觉锁定后 / 内容管道前 + 一句理由 | 是否只回答了「对不对」没回答「何时做」？ |




### 1.7 AI 收工前自检（复杂任务/判断）

- [ ] 是否命中 §1.4 却直接写码？  
- [ ] 是否做了复述 + 完备度声明（复杂任务）？  
- [ ] **多源任务是否先 §1.8 执行计划**（本项目细则 → `MULTI-SOURCE-EXEC-PLAN.md`）？
- [ ] 判断是否至少 2 选项 + 量化表（复杂判断）？  
- [ ] 结论是否写入 STATUS「最近决策」？  
- [ ] **是否把本宪章或 AGENTS 里的复杂任务节删短了？**（禁止）

### 1.8 多源任务执行计划

**规定什么**  
任务同时依赖 **多个文件** 中的要求时，**动手前**必须先外化：读哪些源、优先级谁先谁后、冲突如何裁决。禁止在脑内合并后直写代码。

**适用场景**

- 同时引用 Skill、规范文档、代码、用户既往决策等 **≥2 个权威源**
- 各源可能 **冲突** 或 **层级不清**
- 常与 §1.4 复杂任务、§1.5 复述/信息清单 **同条回复** 输出

**不适用**

- 仅改 **单一** 文件且指令明确（含用户 `@` 唯一路径）
- 无规范冲突的纯机械操作（跑脚本、改 typo）

**使用方式**

1. 输出 **`## 执行计划`**，必含 **参考来源优先级表**：完整路径 · 本任务从该源取什么 · 冲突时服从谁  
2. 可选：执行步骤、硬天花板（摘自高优先级源）、交付物  
3. **默认**在当次回复；跨会话 / 多步任务可落盘项目 `sessions/`  
4. **用户确认后再动手**；已开工发现漏读 → **停止** → 补计划 → 确认 → 继续  
5. **禁止**：只读 Skill 的子文件（如单个 recipe）而声称已执行 Skill 全文

**与 §1.5 的关系**  
优先级表 = 信息清单里「缺什么会做错」的 **核心项**（各源职责与层级）。

**计划最小模板**

```markdown
## 执行计划 · [任务]
### 参考来源与优先级
| 优先级 | 参考文件 | 本任务取什么 | 冲突时 |
| 1 | … | … | 以本行为准 |
### 需确认后再动手
- [ ] 优先级表无误
```

**本项目细则**（demo 优先级、D4 案例、完整模板）→ `docs/MULTI-SOURCE-EXEC-PLAN.md`（见 AGENTS 索引）

## 二、对话整理规则（时间 · 原文 · 最新在上）



### 2.1 背景

- 对话是 **女娲 skill 蒸馏「网站上的 Archie」** 的原始语料。  
- **你的发言**：必须**完整原文**；**AI 回复**：简单**摘要**（长回复会污染语料）。  
- **硬执行层**：`.cursor/rules/project-gotchas.mdc`（`alwaysApply: true`）— 整理对话前从 **transcript JSONL** 提取 `<user_query>` 逐字粘贴；禁止用摘要冒充 Archie 原文。  
- 上下文会被压缩；你说「整理对话」时要**立即落盘**，不能等收工。  
- 称谓固定 **Archie**，不用「用户」。



### 2.2 触发词

命中即归档（不必等收工）：

- `整理对话`  
- `归档语料`  
- `保存这次讨论`



### 2.3 存哪

```
knowledge/raw/dialogues/YYYY-MM-DD-主题.md
```



### 2.4 文件结构（从上到下）

```
---
  YAML 元数据（doc_type: dialogue_corpus）
---
## 修订记录（最新在上）
## 读法说明（一轮 = Archie 一次发言）
## #00N · Archie   ← 最新轮次在最上
[Archie 原文，一字不改]
### Assistant
[一行摘要]
---
## #00(N-1) · Archie
...
```



### 2.5 轮次与时间


| 项             | 规则                                                                                                                        |
| ------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **轮次** `#00N` | 本会话内 Archie 第几次发言；从 `#001` 递增；**禁止**用 Cursor transcript 行号                                                                |
| **读序**        | **最新在最上**（`#010` 在 `#009` 上面）                                                                                             |
| **时间**        | 实时归档：每条用电脑**北京时间，精确到秒**                                                                                                   |
| **补档**        | transcript 无原始时间戳时，在文件头说明；`revised_at` 记补档时刻                                                                              |
| **取时命令**      | `[System.TimeZoneInfo]::ConvertTimeBySystemTimeZoneId((Get-Date), 'China Standard Time').ToString('yyyy-MM-dd HH:mm:ss')` |




### 2.6 排版范例（一轮）

```markdown
## #003 · Archie · 2026-06-30 15:11:50

（此处是 Archie 原文，完整粘贴，不改写）

### Assistant

一行摘要说明 AI 做了什么。
```



### 2.7 对话归档自检

- [ ] Archie 原文是否完整、无摘要替换？  
- [ ] 最新轮次是否在最上？  
- [ ] 轮次是否从 1 递增、无 transcript 行号？  
- [ ] AI 回复是否仅一行？  
- [ ] `speaker: Archie` 是否在 YAML 中？

## 三、新需求覆盖旧需求（含「项目最新说明」体系）



### 3.1 背景

- 新决策出来后，旧 md **看起来像仍有效** → 你和 AI 都会晕。  
- **默认不删文件**（Git 历史人不会翻）；用 **状态 + 权威源指针** 标明谁才是「现在对的」。  
- **最新状态不是堆进一个文件**，而是**一条线一个权威源**。



### 3.2 「项目最新说明」文件指哪些


| 你想知道…        | 权威源                          | 说明                            |
| ------------ | ---------------------------- | ----------------------------- |
| 网站几页、布局、功能   | `docs/SITE-MANUAL.md`        | 给人看；含 ASCII 线框；`[已实现]`/`[规划]` |
| 做到哪、下一步、阻塞   | `docs/STATUS.md`             | 约一页；含进度表                      |
| demo 试吃与视觉混搭 | `docs/DEMO-TASTING-NOTES.md` | 勿假定某一 demo 已锁定                |
| 协作根本规则       | **本宪章**                      | 比建站任务更重要                      |
| 代码真实行为       | `src/`                       | 覆盖一切文档                        |
| 早期产品意图       | `guidelines/`                | **可能过时**；文首有声明                |
| 讨论 WHY       | `docs/sessions/`             | 只增不改                          |




### 3.3 四种处置（覆盖类型）


| 类型         | 含义        | 旧文件                                              | 最新写入              |
| ---------- | --------- | ------------------------------------------------ | ----------------- |
| **A 推翻**   | 方向错了      | 保留 + YAML `status: superseded` + `superseded_by` | 权威源 + STATUS      |
| **B 搁置**   | 暂停，日后可能续做 | 保留 + `status: shelved` + `resume_when`           | STATUS + 原专题 doc  |
| **C 改写**   | 同主题升级     | 原地改权威 doc + 修订记录顶行                               | 同一文件              |
| **D 探索作废** | demo 试完否决 | 保留 demo + STATUS 标注                              | sessions + STATUS |


**仅当**同时满足才可物理删除：临时草稿（`.cursor/plans/`）+ STATUS 已记录 + 权威源已吸收 + Archie 同意或明显垃圾。

### 3.4 冲突时以谁为准（高 → 低）

```
src/  →  SITE-MANUAL  →  STATUS  →  专题 doc  →  guidelines  →  旧 sessions
```

协作规则冲突：**本宪章** → AGENTS → gotchas。

### 3.5 覆盖发生时 AI 执行清单

1. 判定类型（A/B/C/D）
2. 更新 **§3.2 权威源地图** 里对应的那一个
3. 旧文件：YAML `status` + **修订记录表顶行**（`revised_at` 到秒）
4. STATUS「最近决策」加一行：`时间 · 旧→新 · 权威路径`
5. 长讨论 → sessions append delta（不复制 STATUS 整表）
6. `guidelines/` **默认不改**（除非 Archie 明确说更新需求文档）
7. 改 `src/` 页面 → **同步 SITE-MANUAL** 线框



### 3.6 文首 YAML 与修订记录（摘要）

- 会变的项目 md：**YAML frontmatter**（无 `okf:` 嵌套）+ `## 修订记录` **置顶**（最新在上）  
- Preview **显示** YAML 为刻意设计（Archie 也要读 `status` / `revised_at` 等）  
- 字段细则 → `[docs/DOC-FRONTMATTER.md](DOC-FRONTMATTER.md)`  
- 生命周期细则 → `[docs/DOC-LIFECYCLE.md](DOC-LIFECYCLE.md)`  
- `docs/_archive/`：仅当旧**文件名本身会误导**时使用；sessions **永不**进 archive



### 3.7 本项目已发生例子


| 事件               | 类型     | 教训                                  |
| ---------------- | ------ | ----------------------------------- |
| 设计灵感站参考          | A 推翻   | 写入 sessions + STATUS，demo 主线改为案例库试吃 |
| Logo 字体          | B 搁置   | LOGO-FONT-BRIEF + `resume_when`     |
| 知识库立即全建          | A 推翻   | 延后；先视觉混搭                            |
| 无 SITE-MANUAL 时期 | C 缺口   | 新建 SITE-MANUAL 作「给人看的网站真相」          |
| AGENTS 误删复杂任务节   | **事故** | 不得再犯；本宪章 + AGENTS 双份保留              |



### 3.8 对话中的新需求与修改意见（即时同步文档）

**背景**：约束只留在当次对话里 → 上下文压缩或新会话后 AI「记不住」；与 §3.1「新决策只写在对话里」同类问题。

**硬规则**（Archie 2026-07-08 主动要求写入宪章）：

1. **触发**：Archie 在对话中提出**新要求**、**修改意见**、**纠偏**（含对已实现功能的 UX/行为约束），即视为可能覆盖或补充已有 doc。
2. **时机**：**同一轮或紧接下一轮** — 与改代码**同步或先于改代码**落盘；**禁止**只改 `src/` 或只口头答应、等「收工」再补 doc。
3. **写哪**：按 §3.2 权威源地图 — 网站现状 → `SITE-MANUAL` · 进度/决策 → `STATUS` · B 阶段视觉块与验收 → `MIX-MATCH-LIST` · 专题 → 对应 `docs/主题-*.md` · 协作规则 → 本宪章 + AGENTS/gotchas。
4. **怎么写**：YAML `revised_at`（北京时间到秒）+ `## 修订记录` 顶行追加；若**推翻**旧表述 → 旧条目标 `已推翻` 或修订表写「推翻：…」+ 新约束写清（见 [DOC-LIFECYCLE.md](DOC-LIFECYCLE.md)）。
5. **验收标准**：下一会话 AI 只读 STATUS + MIX-MATCH（等索引 doc）即可得到**最新**约束，无需依赖「记得上次聊天」。

**非目标**：不必每句闲聊都写 doc；仅 **会改变怎么做 / 怎么验** 的发言必须落盘。

## 四、不同文件读取与注入优先级



### 4.1 背景：为什么不是「全读一遍」


| 格式                           | 典型                      | 是否每会话自动全文进 AI 上下文                 |
| ---------------------------- | ----------------------- | --------------------------------- |
| `.mdc` + `alwaysApply: true` | gotchas、conventions     | **是**                             |
| `AGENTS.md`                  | 工作区规则                   | **是**                             |
| `docs/*.md`                  | STATUS、本宪章、SITE-MANUAL… | **否**（除非 gotchas/AGENTS 点名或你 `@`） |
| `.agents/skills/*/SKILL.md`  | 花叔、project-context…     | **否**（任务匹配才加载）                    |


**结论**：要「每会话必守」→ 写 **gotchas / AGENTS**；长细则写 **docs/** 并由入口**点名路径**；**本宪章**靠 gotchas + AGENTS 索引，复杂任务时 AI **必须 Read 本文件**。

## 五、文档写作与规则落盘（正式讨论收录）



### 5.1 沟通与书写语言

- **中文为主**；表格、字段名、技术标识可用**中英并列**（如 `status` / 状态）。  
- 给你看的说明（SITE-MANUAL、本宪章）避免堆砌术语；必要术语第一次出现宜有一句白话。



### 5.2 AGENTS 是入口，不是垃圾桶


| 做                                    | 不做                                           |
| ------------------------------------ | -------------------------------------------- |
| AGENTS 放：索引、每会话必守**摘要**、链到本宪章与专题 doc | 把整站 ASCII 线框、长 Gate 表、重复细则全堆进 AGENTS         |
| 细则放：`docs/` 专题文件 + **本宪章**           | 为「保证触发」单独建 `DECISION-PROTOCOL.md`（**已明确不建**） |
| 要每会话触发 → gotchas + AGENTS 点名 + P0 注入 | 把重要规则只写进「不会被读的」存档 doc                        |


**DECISION-PROTOCOL.md**：曾提议存放判断手册细节；你与 AI 商定 **不单独建文件** — 复杂任务/判断写入 **本宪章 + AGENTS + gotchas**，避免多一个 AI 不会自动读的路径。

### 5.3 新增 / 修改协作规则时的落盘顺序

1. **复杂或新类规则** → 先用 §一 四步与 Archie **讨论确定**
2. **先改本宪章**（**通用**规则，遵守 §0.4）
3. 本项目路径/案例/起点表 → 对应 `docs/` 专题（如 `MULTI-SOURCE-EXEC-PLAN.md`）
4. 再改 AGENTS / gotchas（摘要 + **双链**：宪章章节 + 项目细则 doc）
5. STATUS「最近决策」一行（`revised_at` 到秒）
6. 长讨论 → `docs/sessions/` **append delta**（不复制 STATUS 整表）
7. 新建 doc 仅当 **项目细则/域权威** 需要独立维护（如 `MULTI-SOURCE-EXEC-PLAN.md`）；**通用规则不进新 doc**



### 5.4 sessions 与 git 的分工


| 载体                            | 写什么              | 不写什么                     |
| ----------------------------- | ---------------- | ------------------------ |
| `docs/sessions/YYYY-MM-DD.md` | 当日长讨论、多方案、WHY    | **只增不改**；不复制 STATUS 进度整表 |
| `docs/STATUS.md`              | 主任务、进度表、接下来、最近决策 | 协作四步全文（在本宪章）             |
| `git commit`                  | 代码变更             | 不必把 diff 再抄进 sessions    |




### 5.5 新增一条「项目独特规则」决策树


| 内容形态 | 写哪 |

## 八、`.cursor/rules`（mdc）与 `AGENTS.md` 分工



### 8.1 为什么要 `.mdc`

- Cursor 认 frontmatter：`alwaysApply: true` → **每会话全文注入**；`globs:` → 编辑匹配文件时注入。  
- 纯 `.md` 没有这些开关；`docs/` 默认**不会**每会话全文进上下文。



### 8.2 三层放什么


| 层    | 位置                         | 何时进 AI 上下文                        |
| ---- | -------------------------- | --------------------------------- |
| 硬约束  | `project-gotchas.mdc` 等    | **每会话自动**                         |
| 元规则  | `project-conventions.mdc`  | **每会话自动**                         |
| 专题规范 | 本宪章、`DOC-*`、`SITE-MANUAL`… | gotchas/AGENTS **点名**或 Archie `@` |
| 活进度  | STATUS、sessions            | 新任务先读 STATUS                      |
| 稳定需求 | guidelines                 | 可能过时；`@` 或涉及时                     |




### 8.3 你的理解校正


| 你的理解          | 更准确                                  |
| ------------- | ------------------------------------ |
| `.mdc` 强制全文注入 | ✅ 仅当 `alwaysApply: true`（或 globs 匹配） |
| `.md` 不一定全文   | ✅ 对 `docs/`、`guidelines/`            |
| AGENTS 也只有描述  | ⚠️ 本仓库 **AGENTS.md 每会话全文注入**（工作区规则）  |




### 8.4 重要规则如何「随时触发」

1. **每会话必守** → P0：`gotchas` + `conventions` + `AGENTS`（短摘要 + 链宪章）
2. **新任务必 Read** → P1：STATUS、SITE-MANUAL、**本宪章**
3. **细则不堆 alwaysApply** → 避免 token 爆炸；靠 gotchas 一行索引 + 复杂任务时 **Read 宪章**
