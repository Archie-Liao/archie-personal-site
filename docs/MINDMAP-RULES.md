## §1.0 · 「知识卡」和「AI要点」分别是什么？（怎么确认的）

| UI 踢标 | 数据来源（唯一确认方式） | 含义 | **不是** |
|---------|--------------------------|------|----------|
| **知识卡** | `post.knowledgeCards[].front` | 该篇录入的**知识卡片正面词条**（概念/术语标签） | 不是从全文 NLP「挖」出来的；不是 AI 现场编的 |
| **AI要点** | `post.aiSummary.keyPoints[]` | 该篇 **AI 总结里的要点条目**（过滤含「待补」） | 不是知识卡的子概念；无映射时**不声称**从属于某张卡 |

**怎么确认的？**  
只认 `posts.ts`（或将来 md 同源字段）里**已经写好的结构化数据**。  
没有第二套语义引擎；不能回答「为什么这条要点挂在这张卡下」→ **不得挂**（见 §1.4）。

旧称「关键词 / 要点」易与正文用词混淆，界面已改为 **知识卡 / AI要点**。

## §1 · 文案怎么写（中心 / L2 / L3）

### 1.1 中心议题（core）

| 规则 | 说明 |
|------|------|
| 来源 | `post.title` |
| 目标 | **2～10 字**（拉丁可略长）、可读、**单行**、**禁止「…」** |
| 切句 | 按 `，！？：;` 切开，优先完整分句 |
| 压缩 | 只去套话/标点（`lightCompress`）；**禁止**删「过/的」等导致「错过→错」 |
| 反例 | `信息爆炸！别怕错过优质内容` → ❌`信息爆炸别怕错优` → ✅`信息爆炸` |
| 正例 | `手机被忽视，AI正在重蹈覆辙` → ✅`AI重蹈覆辙` |

实现：`extractCoreTopic`。CSS：`.post-mindmap__node--core .post-mindmap__keyword { white-space: nowrap }`。

### 1.2 二级 · 知识卡（L2）

| 规则 | 说明 |
|------|------|
| 来源 | `knowledgeCards[].front`（无卡时才把 AI要点升为 L2 色块） |
| 目标 | **≤8 字**、完整词，禁半截 |
| 视觉 | 每节点独立明亮色底 |

实现：`extractCardKeyword`。

### 1.3 三级 · AI要点（L3）

| 规则 | 说明 |
|------|------|
| 来源 | `aiSummary.keyPoints`（过滤「待补」） |
| 目标 | **≤8 字**；优先「」内短语 |
| 样式 | **无背景色**，字号小于 L2 |

实现：`extractPointKeyword`。

### 1.4 父子关系（有没有「真联系」）

| 状态 | 规则 |
|------|------|
| **当前** | 无显式归属字段时：AI要点 **只挂 `core`**（弱相关虚线），**禁止**轮询挂到知识卡冒充父子 |
| **以后** | 仅当 posts 给出 `parentId` / 人工映射时，才把 L3 挂到对应 L2 |
| **自检** | 不能回答「为什么挂在这张卡下」→ 不得挂 |

## §2 · 层级与视觉

| 层级 | 内容 | 视觉 |
|------|------|------|
| 中心 | 议题短标签 | 无底色、黑体、**单行** |
| **L2** | 知识卡（或升格 AI要点） | **每节点独立明亮色**（`data-swatch` 0–7） |
| **L3** | AI要点 | **无底色**，靠更小字号 |

禁止：同环同色；灰土死莫兰迪（见试吃 Never）。

---

## §3 · 线型

| 关系 | 线 |
|------|----|
| 强（知识卡） | 实线 |
| 弱（AI要点） | 虚线 |

拖动时：锚点按相对位置左右/上下切换；**拖父整枝平移**。  
**core→AI要点（逻辑图）**：必须走 `avoidBandPath` 竖廊拐弯，**禁止**直线穿过知识卡色块。

## §4 · 布局选型（对齐 XMind）

### 4.0 怎么判定用哪种结构？

实现：`chooseMindmapLayout(l2Count, hasL3)`。工具栏文案来自 `layoutReason`。

```
有 AI要点（L3）？ ──是──→ logic 逻辑图
        │
        否
        │
主分支 L2 ≥ 3？ ──是──→ logic 逻辑图
        │
        否
        │
        └──→ map 中心导图（少节点发散）
```

| 判定条件（按优先级） | 结构 | 理由（对齐 XMind） |
|----------------------|------|-------------------|
| 存在 L3（AI要点），或 L2≥3 | **logic** | 有分析层级 / 分支较多 → 左主题、右展开，便于跟读 |
| L2&lt;3 且无 L3 | **map** | 少节点发散 → 中心放射更合适 |

**不是**凭标题语义或文章「简不简单」猜；只数结构化节点。

### 4.0b 整体居中（硬规则）

节点相对关系算完后，对**整组**做包围盒居中（`centerComposition`）：

- 组中心对齐画框 50%/50%
- 过宽/过高时等比微缩，边距 ≥10%
- **禁止**算完后整组贴在左边（旧逻辑图 core≈12% 的问题）

### 4.1–4.3 构图细则

| 条件 | 布局 | 构图 |
|------|------|------|
| 有 L3，或 L2 ≥ 3 | **logic 逻辑图** | 主题相对组在左；见下 |
| 少节点发散 | **map 导图** | 中心放射 |

实现：`layoutLogic` / `layoutMap` → `centerComposition`。画布：logic 用约 16:10。

### 4.1 逻辑图 · 仅知识卡

主题相对左侧 → 知识卡一列（右）→ **再整体居中**。

### 4.2 逻辑图 · 知识卡 + 挂 core 的 AI要点（常见）

**上下分带**（相对组内坐标，居中前）：

| 带 | 内容 | 约 y%（居中前） |
|----|------|----------------|
| 上带 | 知识卡色块 | 12–42 |
| **空廊** | 禁止放节点 | 42–60 |
| 下带 | AI要点（无底） | 60–90 |

中心略偏上，实线进上带、虚线进下带；虚线经近中心竖廊拐入，不穿上带色块；最后整组居中。

### 4.3 防重叠（硬规则）

| 禁止 | 要求 |
|------|------|
| 节点框互相压盖 | 同列纵向最小间距；分带时空廊不得放节点 |
| **线穿过其他节点色块** | core→AI要点 用竖廊避让；不得与知识卡框视觉重叠 |
| 知识卡与 AI要点挤在同一列同一 y 段 | 有两者时必须分带（§4.2） |

**为何曾出现重叠？**  
旧布局把知识卡与 AI要点都放在中→右的同一竖直通道，core→要点的虚线必然穿过中间的色块——属于布局规则缺失，不是「偶尔 bug」。现已用分带 + 避让连线修掉。

未启用（可后加）：组织图、鱼骨、时间轴。

## §5 · 自检清单（改导图 / 改抽取必跑）

1. **命名**：踢标是否为「知识卡 / AI要点」？来源是否分别是 cards / keyPoints？  
2. **中心**：单行？无半截词？  
3. **L2**：异色？词完整？  
4. **L3**：无底色？更小字？  
5. **关系**：说不清归属 → 只挂 core  
6. **重叠**：有知识卡+AI要点时是否上下分带？虚线是否穿色块？（肉眼：上带框内不应有虚线穿过）  
7. **居中**：整组是否大致在画框正中，而非贴左？  
8. **拖动**：整枝跟随；线仍接框边  
9. **回归**：`/post/ep096`、`/post/ep099`

```powershell
node --input-type=module -e "import { extractCoreTopic, buildMindmapModel } from './src/app/utils/mindmapStructure.ts'; import { posts } from './src/app/data/posts.ts'; for (const p of posts) { const m=buildMindmapModel(p); console.log(p.id, m.layout, m.coreTopic, m.branches.map(b=>b.kicker+':'+b.keyword).join(' | ')); }"
```
