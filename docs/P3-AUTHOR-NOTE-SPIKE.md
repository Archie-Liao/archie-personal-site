## 一、页面界面怎么设计（访客 vs 你）

### 1.1 访客（只读）

详情「亲笔旁注」区按块渲染，例如：

```text
┌─ 亲笔旁注 ─────────────────────────────┐
│  〔文字〕                                │
│  今天重读这一期，觉得自己当时太急了……     │
│                                          │
│  〔导图〕  ← 小标签，和文字块分开          │
│     中心：成长消解难题                    │
│       ├ 认知水位变了                      │
│       └ 行动改问题定义                    │
└──────────────────────────────────────────┘
```

- **无旁注**：继续「待亲笔」淡文案（或整节隐藏，尖兵后再定）。  
- **有旁注**：文字用旁注排版；导图块用 **只读画布**（与编辑器同源组件，不可改）。

### 1.2 作者（登录后 · 推荐内联在详情）

详情同页、登录后：

```text
┌─ 亲笔旁注                    [编辑] [导出本篇] ─┐
│  …只读预览…                                     │
└─────────────────────────────────────────────────┘
        ↓ 点编辑
┌─ 亲笔旁注 · 编辑中 ────────── [保存] [取消] ───┐
│  ┌ 文字段落                            [删] ┐   │
│  │  [多行输入框]                            │   │
│  └──────────────────────────────────────────┘   │
│  ┌ 思维导图 · 画布                     [删] ┐   │
│  │  [可点选/拖拽/加子主题的编辑画布]          │   │
│  │  工具：加主枝 · 加子枝 · 删 · 逻辑图/导图 │   │
│  └──────────────────────────────────────────┘   │
│  [+ 文字段落]  [+ 思维导图]                      │
└─────────────────────────────────────────────────┘
```

| 设计原则 | 说明 |
|----------|------|
| **块编辑** | `prose` / `mindmap` 分块；入口「+ 文字」「+ 思维导图」 |
| **导图 = 完整画布一次做完** | **不做**「先表单树、以后再画布」。编辑态即 XMind 式：中心/主枝/子枝、父子连线、拖拽整枝、结构 logic/map（规则见 MINDMAP-RULES / XMind 官方 Topic·Structure·Relationship） |
| **保存明确** | 「保存」写云；取消丢未存草稿 |
| **可选** | `/author` 列表 |

**推翻（2026-07-22）**：曾写「尖兵不做画布拖拽、先表单树」——Archie 要求一次做完，以本节为准。

## 二、怎么区分「文字段落」和「思维导图」？

靠 **数据里的 `type` 字段**，不是靠排版猜。

### 2.1 文档模型（一篇旁注 = 有序块列表）

```json
{
  "postId": "ep099",
  "updatedAt": "2026-07-22T14:21:04+08:00",
  "blocks": [
    {
      "id": "b1",
      "type": "prose",
      "text": "今天重读，觉得当时太急了。\n\n第二段继续写。"
    },
    {
      "id": "b2",
      "type": "mindmap",
      "center": "成长消解难题",
      "children": [
        { "text": "认知水位变了", "children": [{ "text": "同一题权重会变" }] },
        { "text": "行动改问题定义", "children": [] }
      ]
    }
  ]
}
```

| `type` | 你在 UI 上看到 | 写入方式 |
|--------|----------------|----------|
| **`prose`** | 「文字段落」块 | 多行文本框 |
| **`mindmap`** | 「思维导图」块 | **画布编辑** → 存树形 `center` + `children`（可附节点坐标若需还原布局） |

一篇里可以：**只有字、只有图、或交替多块**（评书一段 → 导图 → 又一段闲话）。

云库一行建议：`postId`（主键或唯一索引）+ 整份 JSON（`blocks`）+ `updatedAt`。  
不要把导图拆成另一张表再硬拼——导出和换库更痛。

## 三、能导出吗？换数据库麻烦吗？

**能导出。** 立项时就把「可搬走」写进验收，不绑死 CloudBase。

### 3.1 导出形式：**不是**「一堆没结构的字」

至少提供两种（尖兵可先做 ①，C 再加 ②）：

| 格式 | 长什么样 | 用途 |
|------|----------|------|
| **① JSON（机器真源）** | 上面那种 `postId + blocks[]`；可 `author-notes-export-2026-07-22.json` 含**全部篇** | 换库、脚本导入、备份 |
| **② Markdown 包（人可读）** | 每篇一个 `ep099.md`，块用标题区分 | 丢进 Obsidian / 给 AI / 肉眼翻 |

Markdown 示例（一篇）：

```markdown
---
postId: ep099
updatedAt: 2026-07-22T14:21:04+08:00
---

## 文字

今天重读，觉得当时太急了。

第二段继续写。

## 导图

- 成长消解难题
  - 认知水位变了
    - 同一题权重会变
  - 行动改问题定义
```

### 3.2 换库时你怎么做

```text
CloudBase ──导出 JSON──→ 本地文件 ──导入脚本──→ 新数据库
```

| 保证 | 做法 |
|------|------|
| 不锁死腾讯云 | 业务字段只用通用 JSON，无 CloudBase 私有类型当主结构 |
| 一键备份 | 作者态「导出全部」或云函数定时把 JSON 丢到云存储 / 你下载 |
| 可选进 Git | **归档用** `knowledge/raw/author-notes/` 只存导出快照（备份），**日常编辑仍以云为准**——避免又变成「双真源打架」 |

### 3.3 直接回答你的三句

| 问 | 答 |
|----|----|
| 可以导出吗？ | **可以**，立项验收要包含导出 |
| 换库方便吗？ | **方便**，只要一直用「结构化 JSON」存，换库 = 导出再导入 |
| 导出是一堆文字吗？ | **不是一坨无结构纯文本**；是 **JSON（保类型）+ 可选 Markdown（好读）**。纯 txt 大杂烩不做主格式 |

## 四、推荐架构（防覆盖 · 含块模型）

```text
你在站内改旁注 ──写入──→ CloudBase（真源：postId + blocks JSON）
                              ↑
详情页打开 ←──读取─────────────┘
导出 ←──同一份 JSON / MD────────┘

GitHub push ──只部署──→ 静态前端（不写旁注正文）
```

| 层 | 职责 |
|----|------|
| **真源** | `author_notes`：`postId` + `blocks` + `updatedAt` + `updatedBy` |
| **鉴权** | 仅 Archie |
| **前端** | 只读块渲染；作者态块编辑器 |
| **导出** | 全部 / 单篇 → JSON（必做）· MD（建议） |

### 冲突矩阵

| 场景 | 结果 |
|------|------|
| 网页改旁注后再 push 前端 | ✅ 旁注在云，不盖 |
| 导出 JSON 进仓库当备份 | ✅ 备份；编辑仍以云为准 |
| 误把旁注写回 `posts.ts` 当主库 | ❌ 禁止；代码：**云优先** |

## 五、分期（尖兵）

| 阶段 | 交付 | 验收 |
|------|------|------|
| **A · 数据真源** | 云集合（`blocks`）+ 读 API | 详情显示样例 prose + mindmap |
| **B · 写入 + 画布** | 登录/门闩 + 写 API；**prose 编辑 + mindmap 完整画布**（点选/拖枝/加删主题·对齐 XMind 父子与结构） | 改完刷新仍在；**push 前端后旁注仍在** |
| **C · 体验 + 导出** | 打磨；导出全部 JSON（+ 可选 MD） | 可搬走；换库路径清楚 |

**B 硬验收**：网页改（含画布）→ `git push` → 刷新 → **还在**。  
**画布范围（一次做完，不分期偷懒）**：中心/主枝/子枝、父子连线、拖父带子、logic/map 结构切换；Relationship 跨枝线可同做或紧随。Boundary/Summary 等 XMind 高级件 **非必须**。

## 六、本地 `localhost` 和腾讯云域名：都能改吗？

### 你的理解对不对？

**对。** 两边改的都是 **同一份 CloudBase 旁注**（配置成打同一写接口时）：

```text
localhost:5173  ──保存──┐
                        ├──→ CloudBase author_notes
腾讯云域名站    ──保存──┘
```

所以：**不是**「本地改一份、线上改另一份」；传上云后互相能看见（刷新即可）。

### 那为什么还提「登录固定在一个域名」？

说的是 **登录态（谁已登录）**，不是 **旁注数据存哪**：

| | |
|--|--|
| **数据** | 一个云库；本地写了，线上也能读到 |
| **登录 Cookie / 会话** | 浏览器按**域名**分开：`localhost` 登过 ≠ 腾讯云域名自动已登 |

所以：

- 你可以 **两个站都用** 来改旁注，完全没问题。  
- 只是 **每个域名可能要各登录一次**（或各开一次门闩），不是数据会乱。  
- 之前那句容易听成「只能用线上」——**纠正**：双端可写；线上更适合当你「正式环境已部署好编辑器之后」的常用入口，本地照样能写。

### 一句话

> **改旁注：本地和腾讯云域名都可以，都进同一云库。**  
> 「固定域名」只提醒：登录状态不共用，别以为登了 localhost 线上就免登。
