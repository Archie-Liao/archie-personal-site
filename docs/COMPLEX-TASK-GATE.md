# 复杂任务复述门控

> **完整方法论** → [AI-COLLABORATION-CHARTER.md](AI-COLLABORATION-CHARTER.md) §一  
> 本文件 = **可检查的硬触发细则**（何时必须复述、首条回复长什么样）

## 1. 为什么要「硬」一点

仅靠 AGENTS 里的「自动识别」**不能保证**每轮都复述——模型会跳过。  
本门控用 **关键词 + 固定首段标题**，让你一眼能验：**没出现 `## 复述需求` = 流程未执行**。

**做不到的事（诚实边界）**：Cursor **无法**像编译器一样 100% 阻止 AI 不写复述；硬触发 = **规则 + Hook 提醒 + 你可验收的格式**。

## 2. 触发条件（命中任一 → 启用门控）

### 2.1 任务类型（与宪章 §1.4 一致）

- 架构 / 数据流 / 多系统（网站 + Obsidian + OKF）
- **新增路由或改信息架构**（页面、功能块、导航）
- 不可逆或高返工（内容格式、目录大迁移）
- 需求跨多条线（视觉 + 内容 + 知识库同时动）
- Archie 说：**探讨、评估、规划、矩阵、盘点、确认你真的知道、方案、先写计划**

### 2.2 用户话术关键词（正则粗匹配即可）

```
探讨|评估|规划|矩阵|盘点|复述|确认.*知道|方案|计划|信息架构|IA|页面.*功能|该不该|什么时候做
```

## 3. 命中后：首条回复**必须**以此开头

```markdown
## 复述需求

（用你自己的话：目标 / 约束 / 非目标；3～8 行或表格）

**非目标（本条不做）**：（若有）

---
```

之后才允许：写文件、改代码、给长方案。

**例外**：Archie 明确说「不用复述，直接做」。

## 4. 与复杂任务四步的衔接

| 步 | 门控要求 |
|----|----------|
| 1 复述 | 上节 `## 复述需求` **必填** |
| 2 信息清单 | 复述后同一回复或紧接第二条列出 |
| 3 完备度声明 | 缺关键项 → **只规划不写码** |
| 4 执行 | 小步试点 |

## 5. 分层实现（评估结论）

| 层 | 手段 | 硬度 | 状态 |
|----|------|------|------|
| **L1** | `project-gotchas.mdc` alwaysApply + 本文件 | 每会话注入 | ✅ |
| **L2** | `.cursor/hooks` `sessionStart` 注入提醒 | 每会话开头提醒 | ✅ 见 `.cursor/hooks.json` |
| **L3** | `stop` Hook 检测回复无 `## 复述` 时 followup | 事后纠偏；可能扰人 | ⏸ 暂不启用 |
| **L4** | `preToolUse` 拦 Write/Shell | 真·硬挡；易误杀 | ❌ 不推荐 |

**推荐**：L1 + L2；你用 **「首条有没有 `## 复述需求`」** 验收；没有就回一句「走 COMPLEX-TASK-GATE」。

## 6. Archie 验收清单

- [ ] 首条回复含 `## 复述需求`
- [ ] 有 **非目标**（复杂任务时）
- [ ] 改 IA/路由前先有 **计划或盘点表**，不直接改 `src/`
